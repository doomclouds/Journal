# Journal 今日工作台命令化界面重构设计

## 1. 背景

Journal 当前已经具备可用的核心链路：自然语言输入、真实或本地演示 LLM 整理、日记块编辑、确认保存、LLM 配置与诊断。问题不在功能是否存在，而在主界面仍保留了较多工程调试痕迹：状态、材料、编辑、保存、LLM 入口和诊断信息挤在一层，用户需要自己判断下一步该做什么。

本轮以已认可的 HTML 原型为基准：

`docs/superpowers/specs/2026-05-10-today-workbench-command-surface-prototype.html`

目标是把运行时 UI 重构成“日记纸面优先”的桌面产品界面，而不是继续堆调试面板。

## 2. 设计目标

1. 让主界面第一眼像日记软件：中心是可阅读、可编辑的日记纸面。
2. 让用户不必理解内部状态名：所有状态转换成“下一步”“今日材料”“整理状态”等产品语言。
3. 让原始输入可追溯但不抢主舞台：左侧或右侧展示材料来源、时间、标签和写入关系。
4. 让 LLM 配置与今日工作台视觉统一：同一套圆角、卡片、色板、层级和状态表达。
5. 移除源码编辑与高级源码入口：本轮只保留正常日记区块编辑。
6. 不实现快捷键：菜单可以有命令层结构，但运行时不展示快捷键提示、不注册键盘快捷键。
7. 不实现 AI 风格配置：由于当前 prompt 固定，整理风格只作为静态信息或未来预留展示，不做可保存、可测试、可切换的真实配置项。

## 3. 非目标

- 不恢复 JMF 源码编辑、源码检查器、源码保存接口或高级源码入口。
- 不实现全局快捷键、菜单快捷键或 Electron 原生命令菜单。
- 不新增 AI prompt 风格切换能力。
- 不改变后端日记生成、保存、LLM provider、validator 或本地存储格式。
- 不新增 SQLite、版本快照、删除流、语音输入或安装器能力。
- 不把 HTML 原型逐字照搬；实现应服务于当前 React/Electron 代码结构。

## 4. 推荐方案

采用“快速 UI 重构切片”。

主界面重构成四层：

1. 应用命令层：Electron 原生中文菜单栏 + 真实内容区顶部上下文。
2. 今日上下文层：日期、原始对话、下一步。
3. 日记纸面层：阅读优先，点击块进入编辑，底部继续输入。
4. 今日助手层：下一步、AI 整理摘要、今日材料、整理状态、可用动作。

LLM 配置保留现有功能，但视觉上对齐同一套设计语言：供应商列表、连接信息、API Key、配置来源、诊断和高级参数。整理风格不做真实配置，仅以“当前整理方式”或静态摘要出现。

## 5. 主界面信息架构

### 5.1 应用命令层

运行时页面不再渲染 HTML 原型里的外层设备窗口壳，也不在 React 内容区绘制菜单栏。原型中的 titlebar、窗口控制按钮、外框阴影、圆角窗口和浏览器内模拟菜单只是为了让静态 HTML 在浏览器里展示为桌面应用效果，不进入真实 Electron 内容区。

真实菜单由 Electron 原生 `Menu` 提供：

- 原生菜单：`文件`、`编辑`、`帮助`。
- `文件` 菜单提供 `LLM 配置` 和退出。
- `编辑` 使用系统原生 role，保持输入编辑操作可用。
- `帮助` 提供 `关于 Journal`。
- 菜单项不显示快捷键文字。

React 内容区直接从顶部上下文开始：

- 顶部上下文：`Journal`、当前页面名称、今日状态和 API 健康点。
- LLM 配置入口只保留在原生菜单 `文件 -> LLM 配置`；React 内容区顶部不再提供重复的 LLM 配置按钮或状态入口。
- 当前 LLM 供应商可作为右侧 Today Assistant 元信息展示，例如 `DeepSeek` 或 `Mock`，但不承担配置入口职责。
- 保存日记、重新整理、段落添加仍保留在日记工作台内部按钮，不在本轮硬接到原生菜单，避免“看起来能点但其实没用”的假产品感。

### 5.2 今日上下文层

左侧上下文栏用于回答：

- 今天是哪一天。
- 当前日记状态是什么。
- 原始对话有哪些。
- 原始对话被整理到哪些日记块。
- 当前最合理的下一步是什么。

原始对话采用折叠卡片：

- 默认展开最近一条或第一条。
- 每条显示时间、内容摘要、标签。
- 展开后显示原文和“已用于/已提取”的追溯关系。

如果当前数据还没有可靠的标签或写入关系，首版可以显示时间和原文摘要，追溯关系以后端已有 `rawInputs` 与 draft section 关系为准逐步增强。

### 5.3 日记纸面层

中心区域是主体验：

- 标题为“日记纸面”。
- `只看日记 / 日记 + 助手` 是真实视图切换；只看日记时隐藏右侧助手，让纸面获得更宽阅读空间。
- 文档式纸面承载日记块。
- 默认阅读，点击块上的编辑按钮进入段落编辑。
- 编辑态只显示当前块的 textarea 和 `取消`、`保存修改`。
- `raw-inputs` 对用户显示为“今日材料”，只读，不出现编辑按钮。
- 插入块菜单仍保留在日记编辑区内，作为本轮稳定入口；顶层菜单的插入命令不强行接入。
- 源码相关入口全部移除。

底部输入栏承担继续写作：

- label：`补充今天的自然语言输入`。
- 空状态 placeholder：`今天发生了什么？直接写原话...`
- 已有草稿 placeholder：`补充一句今天的事，或者让 AI 基于当前材料重新整理这一版...`
- 主要按钮：`生成草稿` 或保存日记。
- 次要按钮：`重新整理`，继续沿用现有二次确认逻辑。

### 5.4 今日助手层

右侧助手用于回答“现在该干嘛”和“材料从哪里来”：

- 下一步：显示当前产品状态、建议动作和简短解释。
- AI 整理：显示整理摘要、原始输入数量、日记段落数量、手动编辑数量。
- 今日材料：显示 raw inputs 的时间、标签、摘要和写入关系。
- 今日材料和原始对话正文使用圆角加重引用符号，不使用裸直线 border 充当引用样式。
- 整理状态：显示当前 LLM、结构状态、保存目标。
- 错误/attention：显示可理解的处理提示，不暴露内部异常名作为主文案。

“AI 处理风格”本轮不提供可编辑控件。如果需要出现，只能作为只读状态，例如 `当前整理方式：忠实整理`，不能做成一排可选风格按钮。

## 6. LLM 配置界面

LLM 配置页继续只处理模型供应商和参数，不承担日记下一步工作流。

界面结构：

- 供应商列表：本地演示、OpenAI、DeepSeek、智谱 GLM、Custom。
- 供应商列表和当前 LLM 摘要使用字母圆标，字母必须在圆心，作为快速识别供应商的视觉锚点。
- 连接信息：显示名称、模型、Base URL、API Key。
- API Key：默认隐藏，可用眼睛按钮显示；环境变量来源不显示完整 Key。
- 配置来源：环境变量、本机配置文件、本地演示。
- 最近诊断：连接结果、响应时间、可展开详情。
- 高级参数：temperature、max tokens、timeout、JSON mode。

整理风格：

- 当前代码已有 `stylePreset` 字段，但 prompt 固定。
- 本轮不实现风格切换保存、候选测试、provider 激活联动。
- UI 可以以静态摘要方式显示当前整理方式，不用明确写“未实现”。
- 不让用户误以为点击风格会改变模型输出。

## 7. 状态与错误

运行时状态使用产品语言：

- `empty` -> `待开始`
- `draft/reviewing` -> `可保存` 或 `整理中`
- 本地未保存块编辑 -> `有未保存修改`
- `attention` 或结构校验失败 -> `需要处理`
- `processed` -> `已保存`

错误显示原则：

- 主界面先说明影响：正式日记没有被覆盖，原始表达仍保留。
- 再显示可操作建议。
- 技术细节可以保留在较低层级，不作为标题。

## 8. 响应式策略

桌面宽屏：

- 三列布局：左侧上下文、中心日记纸面、右侧今日助手。
- 中心日记纸面拥有主滚动区域。

中等宽度：

- 左侧上下文和中心日记纸面保持同屏，右侧助手下移到下一行。
- 页面整体允许纵向滚动，避免在小窗口里套出多层滚动条。

窄窗口：

- 单列布局。
- 顺序为：顶部上下文、日记纸面、左侧上下文、今日助手。
- 底部输入仍属于日记纸面的一部分，不拆成独立浮层。
- 避免多重嵌套滚动条。

## 9. 可访问性与交互

- 所有按钮保留明确 `aria-label` 或可读文本。
- 原生菜单不实现额外业务快捷键；React 内容区不再渲染菜单栏按钮。
- 输入框、编辑框、保存按钮在 busy 或本地 dirty 状态下保持现有保护逻辑。
- 禁用状态必须有视觉区别，不只依赖颜色。
- 文案不出现“源码模式”“高级源码”“JMF 编辑器”等用户不需要理解的入口词。

## 10. 测试策略

前端：

- 更新 `App.test.tsx`，断言主界面不出现原型窗口壳、React 内绘制菜单栏和顶部 LLM 配置入口，并覆盖原生菜单 bridge 能打开 LLM 配置。
- 断言不出现快捷键提示，如 `Ctrl+`、`Alt+`。
- 断言不出现源码编辑入口。
- 断言 LLM 配置仍能打开、Key reveal 仍工作、环境变量 Key 仍不泄露。
- 更新 `todayWorkbenchView.test.ts`，覆盖产品状态、section 显示名、材料摘要。
- 更新 `styles.test.ts`，覆盖三列布局、窄屏单列布局、源码样式不存在。

后端：

- 本轮 UI 重构不要求新增后端行为。
- 保留现有 `dotnet test Journal.slnx` 作为回归。

人工/视觉验证：

- 启动 API 和桌面应用。
- 检查宽屏、约 960px、窄窗口三种布局。
- 通过 `文件 -> LLM 配置` 打开 LLM 配置面板，确认与主界面视觉统一。
- 检查无源码入口、无快捷键提示、无可点击 AI 风格切换。

## 11. 验收标准

1. 主界面与 HTML 原型方向一致：有原生中文菜单、左侧上下文、中心日记纸面、右侧今日助手；但不把原型展示用的外层窗口壳或模拟菜单渲染到真实应用内容区。
2. 运行时 UI 不出现源码编辑、高级源码、源码保存相关入口。
3. 运行时 UI 不显示快捷键提示，也没有新增快捷键实现。
4. AI 处理风格不成为真实可编辑配置。
5. 日记块编辑、插入、保存、重新整理、确认保存、原生菜单打开 LLM 配置、Key 显示、provider 测试和启用等已有能力不回退。
6. 小窗口下不会出现截图中那种嵌套滚动条互相挤压的问题。
7. `npm test --prefix apps/desktop`、`npm run build --prefix apps/desktop`、`dotnet test Journal.slnx` 全部通过。

## 12. 自检

- 无占位符或未定义需求。
- 明确排除了源码编辑、快捷键实现和 AI 风格真实配置。
- 设计范围集中在 React/Electron UI 重构，不改变后端数据合同。
- HTML 原型中的快捷键文字只作为原型残留，运行时不照搬。
- 旧计划和归档中出现的高级源码抽屉不作为本轮实现依据。
