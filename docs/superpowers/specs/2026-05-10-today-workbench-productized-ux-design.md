# Journal 今日工作台产品化体验设计

> 状态：设计已确认，待实现计划
>
> 视觉原型：[2026-05-10-today-workbench-productized-ux-prototype.html](./2026-05-10-today-workbench-productized-ux-prototype.html)
>
> 相关背景：Phase 2 今日工作台、Phase 3 JMF 编辑器、Phase 5 真实 LLM 接入、LLM 设置体验优化

## 1. 背景

Journal 的主链路已经能跑通：自然语言输入会被保留，LLM 或 Mock 会整理成 JMF Markdown 草稿，用户可以编辑、校验、确认并写入正式 Markdown。功能可用之后，主界面暴露出新的问题：它越来越像调试面板，而不是一个每天使用的晨间日记产品。

当前主界面的主要体验问题：

- 顶部状态直接展示 `reviewing`、`processed`、`attention` 等内部枚举，用户需要翻译它们才知道下一步。
- 左侧 `Raw inputs`、中间 `JMF 编辑器`、右侧重新整理、确认、文件路径等信息同时出现，视觉中心被工程信息抢走。
- 区块编辑默认像表单，源码模式和 JMF 校验入口过早暴露，容易让普通用户误以为自己在维护一种文件格式。
- 输入框位于右侧 Dock，和日记纸面割裂。用户每天打开时看到的不是“今天先写一句”，而是多个工程面板。
- 窄窗口下多个区域容易各自滚动，存在重新变成多滚动条工作台的风险。

本设计只优化今日工作台的产品体验，不改变后端 JMF、AI Provider、raw input 或正式 Markdown 的核心合同。

## 2. 目标

今日工作台需要从“调试面板”回到“日记纸面优先”的产品界面：

1. 用户第一眼看到的是当天日记和下一步，而不是系统内部状态。
2. 原始输入仍然完整保留，但默认以“今日材料”呈现，不使用 `Raw inputs` 作为首屏语言。
3. JMF 结构继续被系统保护，但 marker、front matter、源码编辑默认进入高级层。
4. 编辑能力保留，但默认阅读态优先。用户点击某一段时，才进入段落级编辑。
5. LLM 重新整理入口留在今日工作流，但不抢占保存和写作主动作。
6. 异常态要告诉用户“为什么不能保存、原始材料是否还在、下一步怎么修”，而不是只显示技术错误。
7. 窄窗口采用单一主要滚动容器，避免 LLM 设置页曾出现的多滚动条问题重演。

## 3. 非目标

- 不重做 LLM 设置面板。
- 不改 JMF 文件格式。
- 不新增云同步、版本快照或删除模型。
- 不引入完整富文本编辑器。
- 不在本阶段实现实时流式生成、语音输入或多日期浏览。

## 4. 设计原则

### 4.1 默认像日记

主界面默认应该像一张可阅读的日记纸面。状态、来源、模型、校验和文件路径都必须降级为辅助信息。用户日常不需要知道当前草稿由哪些 marker 组成。

### 4.2 动作要少

主操作固定为：

- 补充今天
- 重新整理
- 保存日记

其它动作进入右侧“今日助手”或高级抽屉，例如新增可选块、查看完整原始输入、查看源码、查看技术错误。

### 4.3 编辑渐进出现

日记段落默认阅读态。hover 或 focus 时出现“编辑”入口；点击段落后只让该段进入 inline editor。这样保留块编辑的安全边界，但不让整张纸面变成表单。

### 4.4 高级层保留质感

源码模式仍然重要，尤其用于高级修复和验证 JMF。它不再作为默认 tab 常驻纸面顶部，而是进入“高级：JMF 源码”抽屉。源码抽屉使用暗色背景，形成清晰的专业工具层级。

## 5. 信息架构

今日工作台采用四层信息结构：

```text
顶部上下文
  -> 日期、产品化状态、当前 LLM 状态入口

日记纸面
  -> 阅读态日记
  -> 段落级 inline editor

底部主操作
  -> 补充今天
  -> 重新整理
  -> 保存日记

今日助手
  -> 下一步
  -> 今日材料
  -> 整理状态
  -> 高级与技术细节
```

宽屏布局为两列：左侧大纸面，右侧今日助手。窄窗口布局为单列：纸面、底部输入、下一步、今日材料、整理状态、高级源码依次排列。

## 6. 状态语言

主界面不再直接展示后端状态枚举，而是转换为用户能理解的产品状态。

| 产品状态 | 触发条件 | 顶部显示 | 下一步提示 |
| --- | --- | --- | --- |
| 待开始 | 当天没有 raw input，也没有 draft | `待开始` | 写下第一段原始材料 |
| 整理中 | 提交输入或重新整理中 | `整理中` | 等待 AI 整理完成 |
| 可保存 | draft reviewing 且校验通过 | `可保存` | 可以保存，或先微调段落 |
| 有未保存修改 | inline editor 或源码抽屉有本地改动 | `有未保存修改` | 保存段落或取消修改 |
| 需要处理 | draft attention 或 editor validation invalid | `需要处理` | 查看原因、修复、重新整理 |
| 已保存 | 当天正式 entry 已写入 | `已保存` | 可以继续补充，或查看正式文件 |

技术状态仍可在“整理状态”和“高级与技术细节”里查看，但不作为首屏主语言。

## 7. 主要交互

### 7.1 空状态

空状态不显示“还没有可编辑的 JMF 草稿”这类工程提示。默认文案是：

- 标题：`今天先写一句`
- 说明：`不用先想结构。写一段自然语言，Journal 会保留原话，再帮你整理成可确认的日记草稿。`
- 主输入 placeholder：`今天发生了什么？直接写原话...`
- 主按钮：`生成草稿`

如果没有真实 LLM 配置，整理状态说明会提示“未配置 LLM 时使用 Mock 方式整理”。

### 7.2 默认可保存态

当 draft 可确认时：

- 左侧纸面展示完整整理稿。
- 顶部状态显示 `可保存`。
- 底部主操作显示 `补充今天`、`重新整理`、`保存日记`。
- 右侧“下一步”提示：`这篇草稿可以保存。你也可以先点击纸面里的某一段微调。`
- “今日材料”展示 raw input 摘要，可展开查看完整原文。
- “整理状态”展示结构已通过、当前 LLM、保存目标等产品化信息。

### 7.3 段落编辑态

用户点击某个可编辑段落后：

- 只有当前段落变成 inline editor。
- 其它段落保持阅读态。
- 段落内显示 `保存这一段` 和 `取消`。
- 保存段落后调用现有块编辑保存能力，并刷新 editor 状态。
- 有未保存段落时，顶部状态显示 `有未保存修改`，右侧下一步提示先保存或取消段落修改。

必需块和可选块的编辑仍走 JMF section 合同，但默认不显示 section id。section id 只在高级层或测试辅助信息里出现。

### 7.4 重新整理

重新整理仍属于今日工作流，不进入 LLM 设置面板。点击后必须保留现有二次确认语义：

- 明确说明会覆盖当前草稿。
- 明确说明不会影响正式 Markdown。
- 成功后刷新纸面、今日材料和整理状态。
- 失败时进入需要处理态，不覆盖正式 entry。

### 7.5 需要处理态

需要处理态要同时回答三件事：

1. 为什么不能保存。
2. 原始材料是否还在。
3. 下一步怎么修。

默认文案不直接把 `validation_failed` 放在首位，而是翻译为：

- `这篇草稿需要处理`
- `正式日记没有被覆盖，原始表达仍然保留`
- `这通常不是你的输入丢了，而是整理结果没有通过结构检查。`

技术错误放进高级详情，仍保留原始错误文本，便于排障。

### 7.6 高级源码抽屉

源码编辑从默认 tab 调整为高级抽屉：

- 默认折叠，只显示 `高级与技术细节`。
- 展开后使用暗色背景。
- 展示 front matter、JMF marker 和完整 Markdown。
- 提供 `校验结构`、`保存源码草稿`。
- 校验失败时把问题回写到“需要处理态”，但不写入正式日记。

## 8. 文案替换

| 旧语言 | 新语言 |
| --- | --- |
| `Raw inputs` | `今日材料` |
| `JMF 日记编辑` | `日记纸面` 或 `日记编辑` |
| `区块模式` | 默认不显示；进入编辑时叫 `编辑段落` |
| `源码模式` | `高级：JMF 源码` |
| `保存块编辑草稿` | `保存这一段` 或 `保存草稿` |
| `保存源码草稿` | 保留在高级源码抽屉 |
| `reviewing` | `可保存` |
| `attention` | `需要处理` |
| `processed` / `updated` | `已保存` |

## 9. 响应式规则

宽屏 `1280px+`：

- 两列布局：纸面 `minmax(0, 1fr)`，今日助手约 `340px`。
- 底部主操作贴近纸面，不放进右侧助手。

默认桌面 `960px - 1279px`：

- 可保留两列，但减少纸面 padding 和助手宽度。
- 如果内容拥挤，切换为单列优先于制造多个滚动容器。

窄窗口：

- 单列布局。
- 顺序为：顶部上下文、日记纸面、底部输入、下一步、今日材料、整理状态、高级源码。
- 主界面只允许一个主要滚动容器。
- 右侧助手不再独立滚动。

## 10. 测试策略

实现时需要补充以下测试：

- 前端渲染测试：默认状态显示产品化状态，不出现 `Raw inputs`、`reviewing`、`attention` 作为首屏文案。
- 前端交互测试：点击可编辑段落后只该段进入编辑态，保存调用块编辑 API。
- 前端交互测试：源码抽屉默认不显示 JMF marker，展开后显示完整源码编辑能力。
- 前端状态测试：attention draft 显示“需要处理”，并说明正式日记未被覆盖。
- CSS 合同测试：窄窗口下主工作台避免多个核心区域同时 `overflow:auto`。
- 端到端视觉验证：使用典型内容、空状态、需要处理态、窄窗口截图检查不截断、不重叠。

## 11. 验收标准

本设计实现后，用户应该能完成以下判断：

- 今天有没有开始写。
- 现在这篇草稿能不能保存。
- 如果不能保存，为什么不能保存以及怎么修。
- 原始表达有没有被保留。
- 当前用的是哪个 LLM。
- 如果需要高级修复，在哪里展开源码。

同时，用户不应该在默认首屏被迫理解：

- JMF marker。
- YAML front matter。
- 后端状态枚举。
- raw input 存储路径。
- draft 文件路径。

## 12. 实现切分建议

后续实现计划建议拆成四个任务：

1. 产品化状态和文案替换。
2. 今日工作台布局重构：纸面、底部输入、今日助手。
3. 段落级 inline editor 与源码高级抽屉。
4. 空状态、需要处理态、响应式和视觉回归验证。

这四个任务都应保持现有后端合同不变，优先改前端信息架构和交互层。
